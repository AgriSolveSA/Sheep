1. How We Think About Payments

A farmer in the Karoo faces different constraints from a farmer near Cape Town. Rather than a single payment gateway, we need a layered system that offers a clear path:

· Tier 1 (Default): PayFast, Ozow, or SnapScan – standard online gateway for farmers who have internet banking.
· Tier 2 (Backup): MTN MoMo Pay – minimal fee (0.5%), works in rural areas, smartphone optional.
· Tier 3 (Offline): Voucher payment via spaza shop – the farmer pays cash at a local shop, and the shop owner activates a prepaid voucher code that you redeem online.

This way, your platform doesn’t turn anyone away. The goal is to let the farmer pick the method that actually works for them, not the one that’s easiest for you.

---

2. Architecture for the Payment Module

Your payment module should sit between the user and the core calculation engine. A farmer submits their farm data, the engine calculates the report, and then the payment module handles the rest. In practice, this means:

```
Farmer completes form → Payment module checks farmer's preferred method → 
If gateway: redirect to PayFast/Ozow/SnapScan → On success, run model and deliver PDF.
If MoMo Pay: redirect to mobile payment flow → On success, run model and deliver PDF.
If voucher: generate unique voucher code → farmer pays cash at spaza → 
On activation, run model and deliver PDF.
```

To manage all of this, your backend will need a simple database table to track payment status and link completed payments to generated reports. This allows you to re‑send a report if a farmer loses it or to offer discounts for repeat customers.

3. Payment Method Details

💳 3.1 Standard Online Gateways (Tier 1)

Gateway Transaction Fee Monthly Cost Best For
PayFast 3.2% + R1.80 R0 All‑in‑one, supports cards + Instant EFT
Ozow 1.5% (negotiable) R0 Cheapest for Instant EFT, no card needed
SnapScan 2.9% + R1.50 R0 (on basic plan) Customers who have the SnapScan app

All three are zero‑monthly‑fee, so there’s no cost to keep them enabled. They require the farmer to have a bank account and some form of internet access, which is why they are Tier 1.

📱 3.2 Mobile Money & Low‑Tech Options (Tiers 2 & 3)

MTN MoMo Pay is the standout. Over 95% of transactions in SA’s informal sector are still cash‑based. MoMo Pay allows merchants to accept payments via QR code, merchant ID, or payment request at a minimal fee of just 0.5% – significantly lower than most existing services. With over 13 million registered MoMo users in SA, it is rapidly gaining traction in townships and rural areas. A farmer with a basic smartphone can use it, and MTN’s merchant acquisition team provides in‑person onboarding and training.

Voucher‑based payments are the true offline fallback. A farmer buys a voucher from a local spaza shop (paying cash), then enters the voucher code on your website to unlock the report. Spaza shops can sell these vouchers using existing VAS terminals already in place for airtime and electricity. This method requires no bank account, no smartphone, and no internet – just a piece of paper with a code.

💸 3.3 Airtime as Currency (Extreme Case)

For farmers who truly have nothing else, airtime can be converted to value. Vodacom’s VodaPay wallet allows users to transfer airtime credit into a digital wallet, which can then be used to pay for goods and services. The service “Play with Airtime” enables mobile users to convert unused airtime into cash withdrawals – roughly 85% of SA mobile users have unused airtime, with an estimated R1.2 billion in credits going to waste annually.

A simple USSD code (*135#) allows airtime transfer without any internet at all. You could accept airtime as a payment, though this would require manual reconciliation. It is a true last‑resort option, but having it as a possibility sends a strong signal that you will find a way to serve every farmer.

🏦 3.4 PayShap (Instant EFT, Low Fee)

PayShap is SA’s real‑time payment system, usable by anyone with a bank account. Some banks now offer very low fees – Standard Bank recently slashed its PayShap fee to a flat R2, and TymeBank offers PayShap for free. It requires a bank account but no card details. You can integrate PayShap through aggregators like Ozow (which now supports PayShap via its TymeBank partnership). This is a good bridge between Tier 1 and Tier 2.

4. Implementation Roadmap

Start simple, then expand as you validate demand. The following phases assume you already have the core model and user‑input forms built.

Phase 1 – Core Online Gateways (1 week)

Integrate PayFast as your primary gateway. It covers the most farmers with the least development effort. Use PayFast’s “Instant EFT” option for those without cards.

Phase 2 – MoMo Pay Integration (1 week)

Register as an MTN MoMo merchant (no registration fee). Add a payment option to your checkout page. Use MTN’s API documentation or a plug‑and‑play WooCommerce plugin. Test with a few farmers in rural areas to confirm the flow works.

Phase 3 – Voucher System (1–2 weeks)

Build a voucher generation API that creates a unique 12‑digit code per unpaid report. Partner with a spaza shop aggregator like Kazang or Shop2Shop to list your vouchers on their VAS terminals. The farmer pays cash at the shop, the shop activates the voucher, your system receives a webhook, and the report is delivered. This is the most work, but it unlocks the entire offline market.

Phase 4 – Airtime Payment (Optional)

Implement a manual process first: the farmer dials the USSD code, transfers airtime to a dedicated number, sends you the confirmation, and you manually generate the report. If demand is high, build an automated integration later.

5. Risks and Mitigation

Risk Mitigation
Farmers without smartphones Voucher system + USSD‑based airtime conversion.
No bank account Vouchers (cash at spaza) or MoMo Pay.
Voucher fraud (fake codes) Generate long, random codes; log each redemption attempt; limit retries.
MTN MoMo Pay technical issues Keep PayFast as fallback; display a notice if MoMo Pay is temporarily unavailable.
Farmer loses the voucher code Allow resend by email/SMS if they provided a contact number.

---

6. Putting It All Together

Your payment module does not need to be a separate piece of software – it is a set of conditional checks that decide how to handle the checkout. At a high level, your code will:

1. After the farmer submits their data, display payment options filtered by what they can use.
2. If they choose a gateway (PayFast, Ozow, SnapScan), redirect them to the external payment page.
3. If they choose MoMo Pay, redirect them to the mobile payment flow or show a QR code.
4. If they choose voucher, generate a unique code and instruct them to pay cash at a nearby spaza shop.
5. After payment confirmation, run the core model, generate the PDF report, and deliver it by email or direct download.

You will need a simple database table to track pending payments and link them to generated reports. The logic for the core model itself remains untouched – you are only adding a payment step before running it.

---

7. Recommendations

· Do not over‑engineer. Start with PayFast + a manual “pay later” option (bank deposit) and add MoMo Pay next.
· Build a voucher system only after validating demand from farmers in deep rural areas.
· Keep your dependencies minimal – use webhooks from payment gateways, not long‑running background workers.
· Document the “no‑internet” path clearly on your website so farmers know they have options.
